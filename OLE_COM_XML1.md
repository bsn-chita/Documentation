# OLE соединение.
OLE-соединение (Object Linking and Embedding) в 1С позволяет управлять одной информационной базой из другой или из внешнего приложения. Сейчас чаще используется более быстрый механизм COM-соединения, но OLE остается актуальным, если нужно запустить полноценное приложение с интерфейсом.

> OLE-соединение можно использовать для перекачки данных между конфигурациями (очень медленно).

1. Создание объекта. В зависимости от версии 1С используются разные идентификаторы. Например, _V83.Application_ запускает «толстый клиент» 1С:Предприятие 8.3.

2. Строка подключения.
Параметры зависят от типа базы:
    - Файловая база: File="ПутьКБазе"; Usr="Логин"; Pwd="Пароль";.
    - Серверная база: Srvr="ИмяСервера"; Ref="ИмяБазы"; Usr="Логин"; Pwd="Пароль";.

## COM соединение.
COM-соединение это более современный и быстрый способ связи между базами 1С по сравнению с OLE. Главное отличие в том, что оно не запускает полноценное приложение с интерфейсом, а создает легкий программный канал к данным. В терминологии 1С «Внешнее соединение» (External Connection) — это и есть программное название механизма COM-соединения. Внешнее соединение это запуск приложения 1С в специальном режиме. При этом интерфейс не загружается. Это способ перекачки данных, но ресурсов тратиться намного меньше.

Для работы используется объект _V83.COMConnector_.

Ключевые особенности
1. Скорость: Работает значительно быстрее OLE, так как не инициализирует модули интерфейса и экранные формы.
2. Экономия памяти: Потребляет меньше ресурсов сервера или клиентского ПК.
3. Ограничения:
    - Нельзя вызвать методы, связанные с интерфейсом (например, Предупреждение(), ОткрытьФорму()).
    - Код в модуле приложения выполняется в специальном режиме (необходимо проверять доступность процедур).
4. Типы данных: Простые типы (числа, строки, даты) передаются напрямую. Ссылочные объекты (Справочники, Документы) остаются "там" — в вашей базе они будут иметь тип COMObject.

Модуль внешнего соединения.
В дереве конфигурации это отдельный модуль, который выполняется только при таком типе подключения. В нем инициализируются параметры сеанса для COM-запросов. Когда вы заходите через «Внешнее соединение», система не выполняет код из «Модуля управляемого приложения» или «Модуля обычного приложения». Работает только «Модуль внешнего соединения».

Права доступа.
В ролях есть отдельное право «Внешнее соединение». Если оно не отмечено, подключиться через V83.COMConnector под этим пользователем не получится.

Директивы.
Если вы пишете код, который должен работать при COM-подключении, его оборачивают в:
```
#Если ВнешнееСоединение Тогда
    // Этот код увидит только COM-коннектор
#КонецЕсли
```

Если резюмировать технические детали этого «специального режима»:
- Минимум ресурсов: Не загружаются графические библиотеки, формы, стили и картинки. Инициализируется только ядро и прикладные объекты.
- Изоляция кода: Выполняется только Модуль внешнего соединения. Код, прописанный в Модуле управляемого приложения (например, привязанный к старту системы), игнорируется.
- Прямой доступ: Вы работаете с объектами базы (документами, справочниками, запросами) напрямую через точку, как если бы писали код внутри самой базы.
- Лимиты: Из-за отсутствия интерфейса попытка вызвать любой диалог или метод Предупреждение() в этом режиме приведет к ошибке.

Основные отличия OLE от COM
- Интерфейс: OLE запускает копию программы с графической оболочкой (её можно сделать видимой через БазаИсточник.Visible = Истина), тогда как COM работает в фоновом режиме без интерфейса.
- Производительность: COM-соединение работает быстрее, так как не тратит ресурсы на отрисовку окон.
- Функциональность: В COM могут быть недоступны некоторые функции, вызывающие диалоговые окна или специфические методы интерфейса.

```

Процедура COM_СоединениеНаСервере()

	БазаИсточник = Новый COMОбъект("V83.ComConnector"); // COM-объекты поддерживается только в операционных системах Windows
	Попытка
		// Файловая ИБ
		ВнешнееСоединение = БазаИсточник.Connect("File=""/home/user/1C/bases1c/DemoSSl3_1"";USR=""Администратор"";PWD="""";");
	Исключение
	     Сообщить( ОписаниеОшибки());
		 Возврат;
	КонецПопытки;

	Выборка = ВнешнееСоединение.Справочники._ДемоНоменклатура.Выбрать();
	Пока Выборка.Следующий() Цикл
		Сообщить(Выборка.Найменование);

	Запрос = ВнешнееСоединение.NewObject("Запрос");
	Запрос.Текст = "";
```

> Если вы видите в названии объекта слово "Application", то это OLE объект. OLE Application это полноценное приложение (как если бы его запустил пользователь), а COM это маленький объект для общения с другой ИБ. COM работает намного быстрее.

# XML
XML - формат файлов для обмена данными.
Структура XML
```xml
<ИмяЭлемента ИмяАтрибута1="ЗначениеАтрибута1" ИмяАтрибута2="ЗначениеАтрибута2"...>
    Текст ИмяЭлемента
    <ИмяПодчиненногоЭлемент ...>
    ...
    </ИмяПодчиненногоЭлемент>
</ИмяЭлемента>
```
Ограничения:
- Возможен только один корневой тег

XDTO - XML Data Transfer Object (Передача объектов через XML). XDTO есть только в 1С, надстройка над XML.

Выгрузка в XML. Варианты от простого к сложному.

Вариант 1
```xml
<Корневой>
    <Номенклатура Код="00001" Цена="1500">
        Товар1
    </Номенклатура>
    <Номенклатура Код="00002" Цена="2500">
        Товар2
    </Номенклатура>
    <Контрагент Код="00001">
        Василёк
    </Контрагент>
</Корневой>
```

Файл.ЗаписатьОбъявлениеXML() - формируется специальный заголовок. В нем отражается кодировка файла, версия xml. Можно делать и без него, но считается правилом хорошего тона.

XMLСтрока() - "Выгружай по стандарту XML". 1С смотрит тип 1С и преобразует его к типу XML. Преобразует 1С-ные типы к универсальному стандарту.

Файл.ЗаписатьКонецЭлемента() Не указывается какой тег закрывается. Автоматически закрывает последний открытый тег. В XML запрещены пересекающиеся теги. Конец тега не может быть позже начала другого тега (допускается только полная вложенность).

```
&НаСервере
Процедура РаботаСXMLНаСервере()
	Файл = Новый ЗаписьXML;
	Файл.ОткрытьФайл("Путь"+"\Выгрузка.xml");
	Файл.ЗаписатьОбъявлениеXML();
    Файл.ЗаписатьНачалоЭлемента("Корневой");
	Запрос = Новый Запрос;
	Запрос.Текст =
		"ВЫБРАТЬ
		|	_ДемоНоменклатура.Код КАК Код,
		|	_ДемоНоменклатура.Наименование КАК Наименование,
		|	_ДемоНоменклатура.Цена КАК Цена
		|ИЗ
		|	Справочник._ДемоНоменклатура КАК _ДемоНоменклатура
		|ГДЕ
		|	НЕ _ДемоНоменклатура.ЭтоГруппа";
	РезультатЗапроса = Запрос.Выполнить();
	ВыборкаНоменклатура = РезультатЗапроса.Выбрать();
	Пока ВыборкаНоменклатура.Следующий() Цикл
		Файл.ЗаписатьНачалоЭлемента("Номенклатура");
		Файл.ЗаписатьАтрибут("Код", ВыборкаНоменклатура.Код);
		Файл.ЗаписатьАтрибут("Цена", XMLСтрока(ВыборкаНоменклатура.Цена));
		Файл.ЗаписатьТекст(ВыборкаНоменклатура.Наименование);
		Файл.ЗаписатьКонецЭлемента();
	КонецЦикла;
	Файл.ЗаписатьКонецЭлемента();
    Файл.Закрыть();
КонецПроцедуры
```

При закрытии файла Файл.Закрыть() возвращается сформированная строка.

# XML сериализация

XML сериализация это автоматическое преобразование объектов в памяти в XML.

> _ВозможностьЧтенияXML()_ это функция глобального контекста, которая проверяет может ли текущий фрагмент данных в XML-файле быть автоматически преобразован (десериализован) в объект или значение типа "1С Предприятие". Метод считывает тип данных XML из объекта _ЧтениеXML_ и определяет, существует ли в текущей конфигурации 1С соответствующий ему тип (аналогично методу _ПолучитьXMLТип_). Обычно используется в цикле перед вызовом функции _ПрочитатьXML()_. Если метод возвращает _Истина_, значит данные можно прочитать как объект 1С (например, справочник, документ или массив). Метод работает по принципу **проверки метаданных**, а не по детальному сравнению состава всех реквизитов "поле в поле". 1С анализирует заголовок элемента (смотрит на атрибуты узла XML и ищет там указание на тип данных 1С например, _CatalogObject.Контрагенты_). Система проверяет, существует ли в ИБ объект с таким типом (именем). Метод вернет _Истина_ если имя типа совпало (внутри элемента не проверяются на совпадение реквизиты, их количество или названия). Если _ВозможностьчтенияXML_ вернула _Истина_, но при этом структура внутри XML отличается, то: метод _ПрочитатьXML()_ **пропустит лишние поля (которых нет в ИБ)**, для отсутствующих в XML полей объекта 1С останутся **пустые значения**, ошибка возникнет только в случае, если XML схема в принципе нарушена или тип данных не может быть приведен к типу 1С (например, текст вместо даты).

XML сериализация используется при передаче между клиентом и сервером в управляемых формах и в - _Версионирование объектов_.
